# Инфраструктура и запуск

## Docker Compose

Файл `docker-compose.yml` поднимает весь стек одной командой:

```bash
docker compose up --build
```

Поднимаются следующие сервисы:

- `qdrant`
- `redis`
- `clickhouse`
- `mongo`
- `loader`
- `backend`
- `frontend`

## Роль каждого контейнера

### `qdrant`

Векторная база данных. Здесь лежат embeddings абстрактов статей.

### `redis`

Кэш поисковых запросов. Используется только как быстрый слой поверх основного поиска.

### `clickhouse`

Аналитическая база. Хранит таблицу `search_logs`.

### `mongo`

Хранит полные документы статей.

### `loader`

Одноразовый контейнер для первоначальной загрузки данных.

Он:

- ждёт готовности БД
- загружает модель `all-MiniLM-L6-v2`
- генерирует демо-данные или читает реальный датасет
- пишет данные в MongoDB и Qdrant
- создаёт схему в ClickHouse

### `backend`

FastAPI-приложение. Это основной API-сервис проекта.

### `frontend`

React + Vite. Показывает интерфейс поиска и аналитики.

## Почему здесь есть `loader`

Загрузка данных выделена в отдельный сервис по двум причинам:

- это разовая операция, а не часть обычной работы API
- backend должен стартовать быстро, а не ждать повторной загрузки корпуса

Если Qdrant уже содержит точки, `loader` пропускает повторную загрузку, если не передан `--force`.

## Зависимости между контейнерами

`backend` зависит от:

- `qdrant`
- `redis`
- `clickhouse`
- `mongo`
- `loader`

Это нужно, чтобы API не стартовал раньше, чем готовы базы и исходные данные.

`frontend` зависит от:

- `backend`

## Переменные окружения backend

Определяются в `backend/config.py`, а реальные значения подаются через Compose.

Главные настройки:

- `QDRANT_HOST`, `QDRANT_PORT`
- `REDIS_HOST`, `REDIS_PORT`
- `CLICKHOUSE_HOST`, `CLICKHOUSE_PORT`
- `MONGO_HOST`, `MONGO_PORT`
- `MODEL_NAME`
- `REDIS_TTL`

## Как работают healthchecks

У `redis`, `clickhouse`, `mongo` и `backend` настроены healthchecks.

Они нужны, чтобы:

- Compose понимал, когда контейнер действительно готов
- зависимые сервисы не стартовали слишком рано

## Как происходит загрузка демо-данных

Сценарий по умолчанию:

1. `loader` запускает `python scripts/load_data.py --fake ${LOAD_COUNT:-500}`
2. Генерируется набор фейковых статей через `Faker`
3. Для каждой статьи создаётся embedding
4. Метаданные сохраняются в MongoDB
5. Вектор сохраняется в Qdrant

Это удобно для демонстрации проекта без необходимости сразу скачивать большой dataset arXiv.

## Как подключён frontend к backend

Во время разработки frontend использует Vite proxy.

Проксируются запросы вида:

- `/api/search`
- `/api/stats`
- `/api/health`

В Docker-среде proxy target должен указывать на сервис `backend`, а не на `localhost`, потому что внутри контейнера `localhost` означает сам контейнер frontend.

## Локальный запуск без Docker

Проект можно запускать и вручную:

1. Поднять БД через Compose
2. Запустить backend локально
3. Запустить frontend локально

Это полезно для отладки, когда не хочется пересобирать образы после каждого изменения.
