# Backend: FastAPI, поиск и работа с БД

## Общая роль backend

Backend — это центральная точка координации проекта.

Он:

- принимает запросы от frontend
- загружает модель embeddings
- обращается к четырём хранилищам
- собирает итоговый JSON-ответ

Главный файл backend: `backend/main.py`.

## `backend/main.py`

### Что происходит на старте

В функции `lifespan` backend делает три вещи:

1. Загружает модель `SentenceTransformer`
2. Проверяет или создаёт коллекцию в Qdrant
3. Проверяет схему ClickHouse и индексы MongoDB

Это значит, что сервис при старте подготавливает минимальную рабочую среду.

### Почему модель загружается один раз

Модель embeddings тяжёлая по сравнению с обычным кодом, поэтому её невыгодно создавать на каждый запрос.

Она хранится в глобальной переменной `model`, а функция `embed(text)` использует уже загруженный экземпляр.

### Endpoint `/api/search`

Это главный endpoint проекта.

Алгоритм:

1. Обрезается и проверяется входной запрос.
2. Ищется готовый ответ в Redis.
3. Если кэш найден:
   возвращается кэшированный ответ, дополнительно проставляется `cached=True`.
4. Если кэш не найден:
   текст переводится в вектор через `embed`.
5. Вектор отправляется в Qdrant.
6. Из результатов берутся `arxiv_id` и `score`.
7. По этим `arxiv_id` достаются полные документы из MongoDB.
8. Собирается итоговый JSON `payload`.
9. `payload` кэшируется в Redis.
10. Поиск логируется в ClickHouse.

### Почему поиск разбит именно так

Qdrant хорош для поиска по векторам, но в нём не хранятся все поля статьи в удобном виде для UI.

MongoDB, наоборот, хорошо подходит для хранения полного документа статьи.

Поэтому поиск двухшаговый:

- сначала semantic retrieval в Qdrant
- потом metadata lookup в MongoDB

### Endpoint `/api/stats`

Здесь backend собирает аналитику сразу из двух мест:

- из MongoDB берёт общее число статей, статистику по годам и категориям
- из ClickHouse берёт общее число поисков и топ запросов

### Endpoint `/api/health`

Возвращает словарь из булевых значений:

- жив ли Qdrant
- жив ли Redis
- жив ли ClickHouse
- жив ли MongoDB
- загружена ли модель

Это удобно и для Compose healthcheck, и для ручной диагностики.

## Клиентские модули БД

Папка `backend/db/` специально разделяет код по хранилищам. Каждый модуль скрывает детали подключения и предоставляет простой набор функций.

### `backend/db/qdrant_client.py`

Назначение:

- ленивое подключение к Qdrant
- ожидание готовности БД
- создание коллекции
- поиск по векторам
- загрузка точек

Ключевые функции:

- `get_qdrant()` — создаёт singleton-клиент
- `wait_ready()` — ждёт, пока Qdrant начнёт отвечать
- `ensure_collection()` — создаёт коллекцию `arxiv`, если её ещё нет
- `search()` — выполняет nearest-neighbor поиск
- `upsert_points()` — записывает batch точек
- `count_points()` — считает количество точек
- `ping()` — healthcheck

### `backend/db/mongo_client.py`

Назначение:

- подключение к MongoDB
- работа с коллекцией `articles`
- индексы
- аналитические агрегации

Ключевые функции:

- `ensure_indexes()` — создаёт индексы по `arxiv_id`, `year`, `categories`
- `find_by_ids()` — получает документы по списку `arxiv_id`
- `total_articles()` — считает число документов
- `articles_by_year()` — строит статистику по годам
- `articles_by_category()` — строит статистику по категориям
- `ping()` — проверка соединения

### `backend/db/redis_client.py`

Назначение:

- нормализация поискового запроса
- чтение кэша
- запись кэша с TTL

Ключевые детали:

- ключ строится через `_key(query)`
- запрос приводится к нижнему регистру
- лишние пробелы схлопываются

Это сделано, чтобы похожие запросы вроде `Vision Transformer` и ` vision   transformer ` не создавали разные ключи.

### `backend/db/clickhouse_client.py`

Назначение:

- создать таблицу `search_logs`
- вставлять записи о поиске
- возвращать top queries
- считать общее число поисков

Ключевые функции:

- `ensure_schema()`
- `log_search(query, results_count)`
- `top_queries(limit)`
- `total_searches()`
- `ping()`

## `backend/config.py`

Это центральная точка конфигурации backend.

Файл:

- читает `.env`, если он есть
- задаёт адреса БД
- задаёт имя коллекции Qdrant
- задаёт размер вектора `384`
- задаёт TTL Redis
- задаёт allowed origins для CORS

## `backend/scripts/load_data.py`

Это отдельный загрузчик данных.

Он нужен не для обслуживания запросов, а для подготовки индекса.

### Что умеет скрипт

- читать реальный arXiv JSONL
- генерировать fake-данные
- фильтровать статьи по категориям
- вычислять embeddings батчами
- писать данные в MongoDB
- писать точки в Qdrant

### Как устроен pipeline загрузки

1. Выбирается источник данных: файл или fake generator.
2. Данные режутся на батчи по `BATCH_SIZE = 100`.
3. Для батча вычисляются embeddings.
4. В MongoDB выполняется `bulk_write` с `upsert`.
5. Для Qdrant формируется список `PointStruct`.
6. Точки отправляются через `upsert`.

### Почему используется `uuid5`

Функция `stable_point_id(arxiv_id)` делает стабильный ID для точки в Qdrant.

Это важно, чтобы повторная загрузка не создавала новые случайные идентификаторы для той же статьи.

## Почему backend написан именно так

Проект маленький, поэтому здесь нет отдельного слоя сервисов, репозиториев и схем Pydantic на каждую сущность.

Выбран более короткий учебный вариант:

- один входной файл `main.py`
- по одному модулю на каждую БД
- простой procedural flow

Для учебной демонстрации это плюс:

- меньше "служебного" кода
- легче показать преподавателю полный путь запроса
- проще связывать код с архитектурной схемой на слайдах
