# Обзор проекта

## Что делает проект

`ArXiv Semantic Search` — это учебное веб-приложение для семантического поиска по научным статьям arXiv.

Пользователь вводит текстовый запрос в свободной форме, например `attention mechanism for image segmentation`, а система ищет не по точному совпадению слов, а по смысловой близости текста.

## Главная идея архитектуры

Проект специально построен не на одной базе данных, а на четырёх разных NoSQL-решениях:

- `MongoDB` хранит метаданные статей
- `Qdrant` хранит векторы и выполняет семантический поиск
- `Redis` кэширует готовые ответы поиска
- `ClickHouse` хранит поисковые логи и используется для аналитики

Это подход `polyglot persistence`: каждая база данных решает только ту задачу, для которой подходит лучше всего.

## Технологический стек

- Backend: `FastAPI`
- Frontend: `React + Vite`
- Embeddings: `sentence-transformers`, модель `all-MiniLM-L6-v2`
- Развёртывание: `Docker Compose`

## Основной пользовательский сценарий

1. Пользователь открывает страницу поиска.
2. Вводит запрос.
3. Frontend отправляет `GET /api/search`.
4. Backend сначала проверяет кэш в Redis.
5. Если кэша нет, backend эмбеддит запрос в вектор.
6. Вектор уходит в Qdrant, где ищутся ближайшие статьи.
7. По найденным `arxiv_id` backend идёт в MongoDB и получает полные метаданные.
8. Готовый результат кладётся в Redis.
9. Информация о поиске логируется в ClickHouse.
10. Frontend показывает карточки статей.

## Какие данные хранятся в системе

### В MongoDB

Каждый документ статьи выглядит примерно так:

```json
{
  "arxiv_id": "2301.00001",
  "title": "Attention Is All You Need",
  "authors": ["A. Vaswani", "N. Shazeer"],
  "year": 2017,
  "categories": ["cs.LG", "cs.CL"],
  "abstract": "..."
}
```

### В Qdrant

Каждая запись — это точка:

- `id` — стабильный UUID по `arxiv_id`
- `vector` — embedding абстракта размерности 384
- `payload` — короткие метаданные, например `arxiv_id`, `title`, `year`, `categories`

### В Redis

Хранятся готовые ответы поиска:

- ключ: `search:{normalized_query}`
- значение: JSON результата API
- TTL: 1 час

### В ClickHouse

Хранятся логи поисковых запросов:

- `query`
- `results_count`
- `timestamp`

## Какие API есть у backend

### `GET /api/search`

Основной поисковый endpoint.

Возвращает:

- найденные статьи
- число результатов
- флаг `cached`
- время выполнения `query_time_ms`

### `GET /api/stats`

Возвращает:

- общее число статей
- общее число поисков
- топ поисковых запросов
- распределение статей по годам
- распределение по категориям

### `GET /api/health`

Проверяет доступность:

- Qdrant
- Redis
- ClickHouse
- MongoDB
- модели embeddings

## Почему именно такая структура

Если бы всё хранить в одной БД:

- документные данные статей были бы менее удобны
- векторный поиск был бы менее естественным
- кэш пришлось бы организовывать отдельно или считать повторно
- аналитика по логам была бы тяжелее

Поэтому этот проект полезен именно как демонстрация того, что разные типы данных лучше разделять по разным хранилищам.
